# Playbook: Rollback

Этот playbook описывает, как безопасно откатывать неудачные изменения в проекте **landing-factory** без превращения репозитория в хаотичную цепочку срочных фиксов.

## Принцип

Rollback — это штатная управленческая операция восстановления последнего устойчивого состояния, а не аварийная импровизация. Цель rollback состоит не только в том, чтобы убрать дефект со staging или production, но и в том, чтобы сохранить прозрачную историю решения.

## Когда делать rollback

| Сценарий | Действие |
|---|---|
| сломана сборка | немедленно откатить проблемный commit |
| критически нарушен UX или конверсионная логика | откатить и разбирать причину отдельно |
| опубликован неверный wording или оффер | откатить до последней согласованной версии |
| в commit попали лишние файлы | сделать revert и собрать чистый commit отдельно |

## Базовый маршрут

| Шаг | Действие |
|---|---|
| 1 | Найти commit, после которого возникла проблема |
| 2 | Подтвердить, что проблема действительно связана с этим изменением |
| 3 | Выполнить `git revert <commit>` вместо переписывания истории |
| 4 | Проверить локальную сборку после revert |
| 5 | Запушить revert в `main` |
| 6 | Дождаться автодеплоя staging |
| 7 | Зафиксировать rollback в `docs/projects/<slug>/change-log.md` |
| 8 | При необходимости описать вывод в `docs/projects/<slug>/decisions.md` |

## Почему предпочтителен revert-first

В рабочем процессе с общим репозиторием безопаснее отменять изменения через новый прозрачный commit, чем переписывать историю ветки. Это упрощает аудит, диагностику и восстановление контекста после инцидента.

## Минимальный командный сценарий

```bash
git log --oneline
git revert <commit>
npm run build
git push origin main
```

## Что нельзя делать

| Запрещённая практика | Почему это плохо |
|---|---|
| экстренно править файлы вручную без фиксации rollback | теряется история решения |
| смешивать rollback с новой функциональной правкой | становится непонятно, что именно устранило проблему |
| откатывать несколько лендингов одним неразделённым commit | усложняется восстановление и review |

## Результат

Корректный rollback должен оставлять после себя понятную цепочку: какой commit был проблемным, чем он был отменён, какой лендинг это затронуло и в каком документе зафиксирован вывод по инциденту.
