# Playbook: Deploying to Staging

Этот playbook фиксирует стандартный способ обновления staging в проекте **landing-factory**. Текущий рабочий сценарий основан на `git push` в основную ветку репозитория, после чего staging обновляется через настроенный автодеплой.

При этом staging не должен использоваться как среда для каждой промежуточной правки. При низком трафике основной риск для лимитов Netlify создают не посетители, а избыточные сборки, постоянные preview-окружения и тяжёлые ассеты.

## Принцип

Staging не считается обновлённым в момент локального изменения файлов. Он считается обновлённым только после того, как изменение зафиксировано в Git, отправлено в удалённый репозиторий и успешно прошло сборку на стороне хостинга.

## Стандартный маршрут

| Шаг | Действие |
|---|---|
| 1 | Внести изменения в код и связанные документы |
| 2 | Проверить, нужен ли staging-deploy именно сейчас |
| 3 | Проверить локальную сборку |
| 4 | Убедиться, что в commit попали только релевантные файлы |
| 5 | Создать маленький тематический commit |
| 6 | Выполнить `git push origin main` только для согласованного этапа review/publish |
| 7 | Дождаться автозапуска staging-deploy |
| 8 | Проверить staging-страницу после завершения сборки |

## Минимальный командный сценарий

```bash
git status
git add <relevant-files>
git commit -m "Meaningful commit message"
git push origin main
```

## Что обязательно проверить до push

| Проверка | Зачем |
|---|---|
| staging действительно нужен на этом этапе | исключаются лишние сборки и расход лимитов |
| локальная сборка проходит | staging не должен ломаться из-за очевидной ошибки |
| в индекс добавлены только нужные файлы | исключается случайный мусор в репозитории |
| ассеты оптимизированы по весу | уменьшается bandwidth и вес публикации |
| commit-message описывает одно изменение | упрощаются review и rollback |

## Что проверять после push

| Проверка | Почему это важно |
|---|---|
| стартовал ли автодеплой | без этого staging не обновится |
| завершилась ли сборка успешно | failed build блокирует публикацию |
| появился ли новый контент на staging | позволяет отличить проблему деплоя от кэша |

## Ожидаемое время обновления

В нормальном режиме staging обычно обновляется в течение нескольких минут после `push`. Если новая версия не появилась, сначала нужно проверить статус сборки и только потом подозревать проблему в коде или кэше.

## Запрещённые практики

Не следует считать staging актуальным после локальной сборки. Не следует смешивать в одном push несвязанные изменения по нескольким лендингам, если это можно разнести. Не следует деплоить без проверки, какие именно файлы попали в commit. Не следует использовать staging как постоянную рабочую среду для каждой мелкой итерации. Не следует запускать deploy ради одиночной текстовой правки, если её можно объединить с другими изменениями в один релиз.
