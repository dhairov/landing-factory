# ТЗ и детальная инструкция по 3 skill'ам для конвейера создания лендингов ЖК

## Назначение системы

Эта система состоит из **трёх skill'ов**, которые работают последовательно и не дублируют друг друга. Их задача — превращать название девелопера в готовый опубликованный лендинг по конкретному ЖК.

Система должна работать как **конвейер с жёсткими handoff-артефактами**: каждый следующий skill получает не произвольный текст, а строго определённый результат предыдущего этапа.

| Skill | Назначение | Главный выходной артефакт |
|---|---|---|
| **Skill 1** | выбрать проблемный ЖК у девелопера и предложить 3–5 JTBD-сегментов | `market_selection_and_segments.md` |
| **Skill 2** | собрать текстовый прототип лендинга под выбранный сегмент | `landing_blueprint.md` |
| **Skill 3** | реализовать страницу, проверить и опубликовать | `implementation_and_release_report.md` |

## Общий принцип проектирования prompt'ов

Промпты для всех трёх skill'ов действительно лучше писать по принципам из guide по prompt engineering. Это означает, что каждый prompt должен содержать **роль**, **контекст**, **входные данные**, **пошаговый workflow**, **строгий формат результата**, **критерии качества** и **запреты**. Такой подход делает результат более воспроизводимым и уменьшает дрейф между итерациями [1].

Из приложенного гайда следуют несколько особенно важных правил для вашей задачи.

| Принцип | Как применять в skill'ах |
|---|---|
| **Be specific about the output** | у каждого skill должен быть фиксированный output-файл с фиксированной структурой |
| **Use instructions over constraints** | лучше писать «Сделай X в таком порядке», чем длинный список запретов |
| **Provide examples** | для сегментов, блоков лендинга и release-report полезно дать 1–2 примера формата |
| **Use variables in prompts** | нужно явно выделить переменные: `{developer_name}`, `{selected_segment}`, `{blueprint_path}` |
| **Document prompt attempts** | для skill'ов полезно хранить версии prompt'ов и наблюдать, где они дают слабый результат |
| **Role + contextual prompting** | каждому skill нужна чёткая роль и контекст этапа, а не просто команда «сделай» |

> «The clearer your prompt text, the better it is for the LLM to predict the next likely text.» [1]

> «When creating prompts for AI models, it is helpful to provide examples. These examples can help the model understand what you are asking for.» [1]

## Архитектура конвейера

Ниже — оптимальная логика всей цепочки.

| Этап | Вход | Решение пользователя | Выход |
|---|---|---|---|
| **1. Research & Segmentation** | название девелопера | выбрать ЖК и/или один из предложенных сегментов | `market_selection_and_segments.md` |
| **2. Content Blueprint** | выбранный ЖК + выбранный сегмент | утвердить или скорректировать текстовый прототип | `landing_blueprint.md` |
| **3. Build & Release** | утверждённый blueprint | утвердить публикацию | код, deploy, release report |
| **Cross-cutting: Deployment Readiness & Cost Control** | изменения, готовые к публикации | подтвердить, что deploy действительно нужен сейчас | решение о типе deploy, проверка ассетов, контроль лимитов |

## Skill 1. Отбор проблемного ЖК и JTBD-сегментов

### Цель

Skill 1 получает **название компании-застройщика** и должен по открытым источникам определить, какой из ЖК этого девелопера сейчас является наиболее проблемным с точки зрения продаж, если сравнивать его с другими активными проектами того же девелопера с учётом стадии продаж и срока сдачи.

После этого skill формирует **3–5 гипотез сегментов потенциальных клиентов** именно для выбранного ЖК и для каждого сегмента даёт draft job-to-be-done формулировки.

### Источники, которые skill обязан использовать

| Тип источника | Что искать |
|---|---|
| **ЕРЗ / наш.дом.рф / GDRF** | перечень проектов, стадия реализации, срок сдачи, базовые параметры проекта |
| **Официальный сайт девелопера** | список ЖК, позиционирование, цены, акции, ход строительства |
| **Сайты конкретных ЖК** | оффер, планировки, аудитория, УТП, темп коммуникации |
| **Открытые витрины / агрегаторы / карточки объектов** | наличие активных продаж, цена, остатки, рыночный контекст |
| **Новости и PR-материалы** | сигналы об ускорении или замедлении продаж, запуск очередей, скидочные механики |

### Обязательная логика работы

Skill 1 не должен просто «выбрать интересный ЖК». Он должен пройти через явную аналитическую последовательность.

| Шаг | Что делает skill |
|---|---|
| **1** | принимает название девелопера и, если нужно, уточняет юридическое/брендовое написание |
| **2** | собирает список активных или недавно активных ЖК девелопера |
| **3** | исключает нерелевантные объекты: полностью распроданные, архивные, неоткрытые |
| **4** | сравнивает оставшиеся ЖК между собой по срокам сдачи, стадии, витрине продаж и признакам темпа реализации |
| **5** | выбирает ЖК, который вероятнее всего отстаёт по темпам продаж относительно внутренних аналогов |
| **6** | формирует 3–5 гипотез целевых сегментов для этого ЖК |
| **7** | для каждого сегмента формулирует draft JTBD: контекст, работа, триггеры, барьеры, желаемый исход |
| **8** | рекомендует один сегмент как лучший кандидат для первого тестового лендинга |

### Критерии выбора проблемного ЖК

Чтобы skill не принимал решение интуитивно, критерии нужно явно зафиксировать.

| Критерий | Как оценивать |
|---|---|
| **Статус продаж** | ЖК должен находиться в активных продажах |
| **Срок сдачи** | учитывать, насколько близко проект к сдаче и как это обычно влияет на темп |
| **Внутреннее сравнение** | сравнивать только с другими ЖК этого же девелопера |
| **Признаки замедления** | слабее представлен оффер, меньше сигналов дефицита, активнее используются скидки, менее убедительное позиционирование |
| **Качество витрины** | сайт, карточки, акционные сообщения, наличие явной коммуникации спроса |
| **Уровень уверенности** | skill обязан отметить, где вывод твёрдый, а где гипотетический |

### Формат результата Skill 1

Skill 1 должен создавать файл `market_selection_and_segments.md` по строгому шаблону.

| Раздел | Содержание |
|---|---|
| `developer` | название компании |
| `source_log` | список использованных источников |
| `active_projects_shortlist` | краткий список активных ЖК |
| `comparison_table` | сравнительная таблица по ЖК |
| `selected_project` | выбранный проблемный ЖК |
| `selection_rationale` | краткое объяснение выбора |
| `segment_hypotheses` | 3–5 сегментов |
| `recommended_segment` | какой сегмент тестировать первым |
| `confidence_and_gaps` | где данные сильные, а где слабые |

### Формат описания сегментов

Для воспроизводимости каждый сегмент должен описываться единообразно.

| Поле | Содержание |
|---|---|
| `segment_name` | узнаваемое человеческое название сегмента |
| `life_context` | жизненная ситуация |
| `draft_job` | главная задача клиента |
| `triggers` | 2–3 вероятных триггера |
| `barriers` | 2–4 барьера к покупке |
| `desired_outcome` | какого состояния хочет человек |
| `landing_angle` | какой угол атаки для лендинга можно тестировать |

### Prompt-структура для Skill 1

Ниже — рекомендуемый каркас prompt'а.

```md
# Role
Ты — аналитик девелоперского рынка и стратег сегментации для лендингов жилых комплексов.

# Task
Получив название девелопера `{developer_name}`, собери по открытым источникам активные ЖК компании, сравни их между собой и выбери тот ЖК, который вероятнее всего отстаёт по темпам продаж относительно других ЖК этого же девелопера с учётом срока сдачи и стадии реализации.

После выбора ЖК предложи 3–5 гипотез сегментов потенциальных клиентов и для каждого сегмента сформулируй draft job-to-be-done.

# Workflow
1. Уточни, о каком девелопере идёт речь, если название неоднозначно.
2. Собери список активных ЖК из открытых источников.
3. Исключи нерелевантные проекты.
4. Сравни ЖК между собой.
5. Выбери один целевой ЖК и объясни выбор.
6. Сформируй 3–5 сегментов.
7. Рекомендуй один сегмент как лучший кандидат для тестового лендинга.
8. Сохрани результат строго по заданному шаблону.

# Output
Верни результат в формате `market_selection_and_segments.md`.

# Quality bar
Не выдумывай недоступные числа. Если данных не хватает, отмечай это явно. Не переходи к текстам лендинга. Не предлагай больше одного целевого ЖК в финальной рекомендации.
```

## Skill 2. Сборка текстового прототипа лендинга

### Цель

Skill 2 получает **выбранный ЖК** и **один выбранный сегмент** из Skill 1. Его задача — собрать не код, а **текстовый master-документ**, по которому потом можно разработать страницу.

Очень важно, чтобы этот skill не писал «просто хороший копирайт». Он должен производить **content blueprint**, пригодный для handoff в разработку.

### На что должен опираться Skill 2

В качестве референса видно, что хороший лендинг уже строится не как абстрактный рекламный текст, а как последовательность блоков: триггеры, hero, aha-моменты, доказательства, сравнение с альтернативами, сценарий после клика и финальный CTA [2].

| Структурный принцип | Что это даёт |
|---|---|
| **Триггеры до hero** | быстро включает релевантный жизненный контекст |
| **Оффер как решение** | связывает проект и жизненную задачу клиента |
| **Aha-moments** | закрывают барьеры и сомнения |
| **Социальные доказательства** | добавляют доверие и культурную валидность |
| **Сравнение с альтернативами** | помогает отстроиться от других сценариев выбора |
| **Что происходит после клика** | снижает тревогу перед конверсией |

### Обязательная логика работы

| Шаг | Что делает skill |
|---|---|
| **1** | принимает выбранный ЖК и выбранный сегмент |
| **2** | извлекает из Skill 1: контекст сегмента, draft job, триггеры, барьеры, желаемый исход |
| **3** | формулирует угол атаки лендинга и главный оффер |
| **4** | собирает структуру блоков страницы |
| **5** | пишет тексты по блокам |
| **6** | задаёт tone of voice и стилевые правила |
| **7** | перечисляет нужные proof points и asset needs |
| **8** | сохраняет всё в `landing_blueprint.md` |

### Формат результата Skill 2

| Раздел | Содержание |
|---|---|
| `project_context` | ЖК, девелопер, выбранный сегмент |
| `segment_summary` | краткое напоминание, для кого лендинг |
| `jtbd_core` | главная логика задачи клиента |
| `offer` | ключевой оффер страницы |
| `tone_and_style` | стилистика, допустимые и запретные ходы |
| `page_structure` | перечень блоков в порядке показа |
| `copy_by_block` | тексты по каждому блоку |
| `proof_points` | что должно быть подтверждено фактами |
| `cta_logic` | основные и вторичные CTA |
| `asset_needs` | какие визуалы и медиа нужны |
| `implementation_notes` | что важно учесть при разработке |

### Рекомендуемая структура `landing_blueprint.md`

| Блок | Описание |
|---|---|
| **1. Trigger bar** | 2–3 коротких триггера над hero |
| **2. Hero** | headline, subheadline, CTA, visual direction |
| **3. Why this project** | почему этот ЖК релевантен именно этому сегменту |
| **4. Aha-moments** | 3 ключевых барьера и их снятие |
| **5. Social proof** | сигналы доверия, отзывы, культурные маркеры |
| **6. Alternative comparison** | чем ЖК лучше других сценариев выбора |
| **7. Offer mechanics** | что человек получает после заявки |
| **8. Final CTA** | повтор оффера и мягкое завершение |

### Правила стиля для Skill 2

### Сквозное правило: Deployment Readiness & Cost Control

Для всей фабрики лендингов нужно явно учитывать не только качество текста и реализации, но и стоимость публикационного контура. При низком трафике основной риск исчерпания лимитов связан не с реальными посетителями, а с внутренним процессом: лишними сборками, preview-deploy, тяжёлыми ассетами и избыточным количеством отдельных окружений.

| Проверка | Вопрос перед deploy |
|---|---|
| **Deploy necessity** | действительно ли нужен новый публичный deploy именно сейчас |
| **Environment choice** | нужен staging или достаточно локальной проверки |
| **Release batching** | можно ли объединить несколько правок в один релиз |
| **Asset weight** | оптимизированы ли изображения и исключены ли тяжёлые медиа |
| **Project topology** | нужен новый проект, или лендинг можно выпустить внутри существующей схемы |

Из этого следует операционное правило: промежуточные итерации по тексту, структуре и визуалам по умолчанию проходят локально, а публичный deploy используется только на этапах согласования и финальной публикации.

### Правила стиля для Skill 2

| Область | Правило |
|---|---|
| **Тон** | B2C, бережный, небанальный, без грубого sales-нажима |
| **Язык** | простой, точный, без штампов вроде «комфорт и качество жизни» |
| **Оффер** | должен быть привязан к жизненной задаче, а не только к метрам и ипотеке |
| **Доказательства** | не выдумывать отзывы и цифры, если они не подтверждены |
| **Сегмент** | нельзя незаметно менять сегмент или перепридумывать ЖК |

### Prompt-структура для Skill 2

```md
# Role
Ты — стратег лендингов и редактор смысловой упаковки жилых комплексов.

# Task
Получив выбранный ЖК `{project_name}` и выбранный сегмент `{selected_segment}`, создай текстовый `landing_blueprint.md`, по которому можно разработать лендинг.

# Context
Используй результаты из `market_selection_and_segments.md`. Не меняй выбранный ЖК и сегмент без явного запроса пользователя.

# Workflow
1. Кратко пересобери контекст сегмента.
2. Сформулируй главный оффер.
3. Собери структуру лендинга.
4. Напиши тексты по блокам.
5. Задай tone of voice.
6. Перечисли proof points и asset needs.
7. Подготовь implementation notes.

# Output
Верни `landing_blueprint.md` в фиксированной структуре.

# Quality bar
Не пиши код. Не уходи в банальные формулировки. Не имитируй факты, которых нет в материалах. Пиши так, чтобы документ был пригоден для handoff разработчику.
```

## Skill 3. Разработка, проверка и публикация страницы

### Цель

Skill 3 получает утверждённый `landing_blueprint.md` и превращает его в рабочую страницу внутри репозитория и сайта. Это уже не исследовательский и не редакторский skill, а **production skill**.

### Обязательная логика работы

| Шаг | Что делает skill |
|---|---|
| **1** | читает `landing_blueprint.md` и проектные правила |
| **2** | создаёт или обновляет страницу по slug проекта |
| **3** | подключает тексты, секции, CTA и нужные ассеты |
| **4** | проверяет route, alias, формы, аналитику и базовую адаптивность |
| **5** | выполняет deploy |
| **6** | возвращает release report |

### Ограничения Skill 3

| Что можно | Что нельзя |
|---|---|
| реализовывать страницу по blueprint | самовольно менять сегмент, оффер или стратегию |
| уточнять технические детали | переписывать смысловую архитектуру без подтверждения |
| сообщать о невозможности реализации отдельных блоков | молча выбрасывать важные блоки |

### Формат результата Skill 3

| Раздел | Содержание |
|---|---|
| `implementation_summary` | что именно было собрано |
| `files_changed` | список созданных/изменённых файлов |
| `route_and_aliases` | canonical route и публичные адреса |
| `forms_and_analytics` | что подключено и как проверено |
| `deploy_result` | staging / production URL |
| `qa_checklist` | адаптивность, CTA, форма, тексты, базовые проверки |
| `known_limitations` | что осталось ограничением |

### Prompt-структура для Skill 3

```md
# Role
Ты — frontend/product implementation specialist для landing-factory.

# Task
Получив утверждённый `landing_blueprint.md`, реализуй страницу в проекте, подключи нужную инфраструктуру, выполни deploy и подготовь release report.

# Workflow
1. Прочитай blueprint.
2. Определи slug, route и структуру файлов.
3. Реализуй страницу.
4. Проверь CTA, форму, аналитику, маршруты и адаптивность.
5. Выполни deploy.
6. Верни `implementation_and_release_report.md`.

# Output
Верни итоговый отчёт и ссылку на опубликованную страницу.

# Quality bar
Не переписывай стратегию лендинга. Если blueprint противоречив или неполон, остановись и запроси уточнение вместо самовольной переработки.
```

## Общие guardrails для всей системы

| Правило | Зачем оно нужно |
|---|---|
| **Каждый skill работает только в своей зоне ответственности** | чтобы не было дублирования и размывания решений |
| **Каждый этап создаёт один главный артефакт** | чтобы handoff был воспроизводимым |
| **Каждое важное решение явно фиксируется** | чтобы следующий этап не «додумывал» смысл |
| **Нельзя молча менять объект, сегмент или оффер** | чтобы не ломать цепочку принятия решений |
| **Если данных не хватает — skill должен это явно отмечать** | чтобы не плодить ложную уверенность |

| Артефакт | Где хранить |
|---|---|
| `market_selection_and_segments.md` | `docs/projects/{slug}/` |
| `landing_blueprint.md` | `docs/projects/{slug}/` |
| `implementation_and_release_report.md` | `docs/projects/{slug}/` |
| ассеты проекта | `public/assets/{slug}/` |
| страница | `src/pages/{slug}/` или по принятой системе роутинга |

## Как сделать систему ещё сильнее

Я рекомендую не просто написать три prompt'а, а оформить их как **отдельные skill'ы с контрактами**. Тогда у каждого skill будет:

| Компонент | Что внутри |
|---|---|
| `SKILL.md` | purpose, input, output, workflow, guardrails |
| prompt-template | основной prompt с переменными |
| output template | шаблон итогового markdown-файла |
| examples | 1–2 хороших примера результата |
| evaluation checklist | как понять, что skill сработал качественно |

Это соответствует лучшим практикам prompt engineering: использовать примеры, документировать попытки, быть точным в output и проектировать prompt как управляемый интерфейс, а не как свободную просьбу [1].

## Моя финальная рекомендация

Оптимальная реализация для вас сейчас выглядит так:

| Приоритет | Что делать |
|---|---|
| **1** | зафиксировать 3 skill-контракта в отдельных `SKILL.md` |
| **2** | для каждого skill сделать output template |
| **3** | для Skill 1 отдельно продумать шкалу confidence и критерии выбора «проблемного ЖК» |
| **4** | для Skill 2 закрепить единый формат `landing_blueprint.md` |
| **5** | для Skill 3 жёстко запретить перепридумывать стратегию лендинга |

Так система получится не просто умной, а **операционно надёжной**.
