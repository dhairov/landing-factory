# Brief: 745-kvartal

Этот документ фиксирует краткий паспорт проекта `745-kvartal` как управляемого артефакта в репозитории. Его задача — дать следующему исполнителю или владельцу проекта быстрый и точный контекст без необходимости восстанавливать историю из переписки.

## Project Summary

`745-kvartal` — это демонстрационный лендинг по переупаковке девелоперского продукта. Он одновременно решает две связанные задачи: показывает, как может выглядеть более сильная упаковка жилого комплекса для конечного покупателя, и служит примером для топ-менеджмента девелопера, которому демонстрируется потенциал смысловой и конверсионной переработки позиционирования.

## Audience Model

У проекта есть двухслойная аудиторная логика, и это должно оставаться явно зафиксированным в документации и коде.

| Layer | Audience | Role in page |
|---|---|---|
| B2C layer | Потенциальный покупатель ЖК | Основной смысловой и эмоциональный адресат hero и большей части лендинга |
| B2B layer | Топ-менеджер девелопера | Адресат финального блока с формой и управленческим CTA |

## Core Scenario

Основной сценарий страницы строится как контраст между текущим рыночным способом упаковки объекта и более сильной, сегментно точной интерпретацией того же продукта. Пользователь должен увидеть, что проблема продаж может лежать не только в самом объекте или рынке, но и в слабом позиционировании, неточной работе с аудиторией и недостаточно убедительной подаче ценности.

## Messaging Boundary

Важное архитектурное правило проекта состоит в том, что B2C- и B2B-слои нельзя смешивать в одном и том же смысловом центре страницы.

| Zone | Rule |
|---|---|
| Hero and main narrative | Ориентированы на потенциального покупателя ЖК |
| Final form block | Ориентирован на топ-менеджера девелопера |
| Cross-over logic | B2B-смысл допускается как мета-рамка проекта, но не должен ломать B2C-оффер страницы |

## Current Constraints

| Ограничение | Значение для проекта |
|---|---|
| GitHub is the source of truth | Существенные решения должны фиксироваться в репозитории, а не только в переписке |
| Staging deploy follows git push | Публикация происходит после push и успешного автодеплоя |
| Documentation must stay project-scoped | Проектные артефакты должны жить по `slug`, а не размазываться по root-уровню |
| Assets must be traceable | Все важные визуальные материалы должны быть перечислены в `assets.md` |
