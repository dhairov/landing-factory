# Корректировки процесса публикации для экономии лимитов Netlify

## Контекст

Для текущего контура лендингов ожидается очень низкий внешний трафик — порядка **5–10 визитов на страницу в месяц**. Это означает, что основной риск исчерпания лимитов связан не с реальными посетителями, а с внутренним процессом: частыми деплоями, избыточными preview-сборками, тяжёлыми ассетами и неэкономной схемой публикации.

## Главный принцип

При таком объёме посещаемости процесс должен быть оптимизирован не под масштаб внешнего спроса, а под **минимизацию build usage и служебного bandwidth**.

## Что меняем в процессе

| Область | Было / риск | Корректировка |
|---|---|---|
| Деплой каждой правки | частые автоматические сборки на каждое изменение | деплой только для согласованных этапов: `draft`, `review`, `publish` |
| Preview deploys | каждый PR или ветка могут создавать отдельную сборку | отключить или резко ограничить preview deploys для лендинговых проектов |
| Staging-сайты | staging может жить постоянно и расходовать лимиты так же, как production | использовать staging только точечно, на период согласования, а не как постоянную среду |
| Ассеты | тяжёлые изображения и видео съедают bandwidth | ввести обязательную оптимизацию изображений и запрет на тяжёлые видеофайлы в прямой публикации |
| Структура проектов | много отдельных сайтов = много отдельных deploy-процессов | по возможности публиковать лендинги в рамках единой фабрики, а не как полностью разрозненные проекты |
| Частота обновлений | мелкие правки сразу уходят в деплой | накапливать пакет правок и выкатывать одним релизом |

## Рекомендуемый процесс публикации

### 1. Локальная разработка по умолчанию

Все текстовые, структурные и визуальные правки сначала проходят локально. Публичный deploy не используется как рабочая среда для промежуточных итераций.

### 2. Один deploy на этап согласования

Промежуточная логика должна быть такой:

| Этап | Правило |
|---|---|
| `draft` | работа локально, без публичного deploy |
| `internal review` | при необходимости один staging-deploy для сверки |
| `client review` | отдельный deploy только когда версия действительно готова к просмотру |
| `publish` | production deploy только после согласования |

Это означает, что лендинг не должен деплоиться на каждую правку копирайта, ассетов или верстки.

### 3. Ограничение preview-окружений

Для лендинговой фабрики стоит отказаться от автоматического создания preview для каждой ветки, если это не критично для команды. Preview-окружения имеют смысл только для сложных изменений, а не для каждой текстовой итерации.

### 4. Жёсткая политика по ассетам

Нужно считать, что именно медиа чаще всего создают лишний расход даже при низком трафике.

| Тип ассета | Правило |
|---|---|
| Hero-изображения | только сжатые версии, без избыточного разрешения |
| Галереи | ограниченное число изображений |
| Видео | не хранить тяжёлые видео напрямую в публикации, если без них можно обойтись |
| PDF/презентации | выдавать по запросу, а не грузить в первый экран |

### 5. Пакетные релизы вместо микродеплоев

Даже при небольшом объёме работы критично перейти от реакции на каждую правку к пакетному релизному ритму. Один согласованный релиз лучше, чем серия мелких deploy-событий.

## Операционные правила для команды

| Правило | Смысл |
|---|---|
| Не деплоить текстовые правки по одной | сокращает количество сборок |
| Не держать staging постоянно активным без необходимости | уменьшает фоновый расход |
| Перед публикацией сжимать ассеты как обязательный шаг | снижает bandwidth |
| Проверять, нужен ли отдельный сайт для нового лендинга | уменьшает количество независимых deploy pipelines |
| Хранить этапность согласования в документах, а не в множестве публичных ссылок | уменьшает зависимость процесса от Netlify |

## Предлагаемая поправка в фабричный процесс

В workflow лендингов нужно явно добавить отдельный контрольный этап **Deployment Readiness & Cost Control**.

Этот этап должен проверять:

| Проверка | Вопрос |
|---|---|
| Deploy necessity | действительно ли нужен новый публичный deploy именно сейчас |
| Asset weight | оптимизированы ли изображения и исключены ли тяжёлые медиа |
| Environment choice | нужен ли staging, или достаточно локальной проверки |
| Release batching | можно ли объединить правки в один релиз |
| Project topology | нужен ли новый проект в Netlify, или можно опубликовать внутри существующей схемы |

## Вывод

При трафике **5–10 визитов на страницу в месяц** экономия лимитов достигается не маркетинговыми ограничениями, а дисциплиной процесса. Основной фокус должен быть на сокращении числа сборок, уменьшении числа превью-окружений, контроле ассетов и переходе к пакетным релизам.
