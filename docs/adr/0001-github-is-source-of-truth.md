# ADR 0001: GitHub Is the Source of Truth

## Status

Accepted

## Context

Проект **landing-factory** развивается как система, в которой одновременно растут код лендингов, проектная документация, история правок, контентные решения и операционные playbook-документы. Без формального правила источник правды быстро размывается между чатом, локальными заметками, временными файлами, браузерными артефактами и репозиторием.

На малом масштабе такая размытость кажется терпимой, но при росте числа лендингов и итераций она приводит к трём системным проблемам. Во-первых, команда перестаёт понимать, где лежит актуальная версия решения. Во-вторых, rollback превращается в угадывание того, какое состояние считать последним согласованным. В-третьих, новый исполнитель не может надёжно восстановить контекст без доступа к истории переписки.

## Decision

В проекте **landing-factory** единственным официальным источником проектной правды считается GitHub-репозиторий и его основная история изменений.

Это решение означает следующее.

| Область | Правило |
|---|---|
| Код | актуальное состояние кода определяется содержимым репозитория |
| Документация | проектная документация хранится в `docs/` и versioned вместе с кодом |
| История изменений | значимые изменения фиксируются через commit history и проектные документы |
| Временные материалы | локальные заметки, черновики и служебные файлы не считаются источником правды |

Любая информация, необходимая для поддержки, передачи, review, rollback или развития лендинга, должна быть либо уже сохранена в репозитории, либо перенесена туда в ближайшей осмысленной итерации.

## Consequences

Принятое решение даёт репозиторию роль не только хранилища кода, но и операционной памяти проекта.

| Положительное последствие | Практический эффект |
|---|---|
| единая точка истины | меньше споров о том, какая версия актуальна |
| связка docs + code + history | проще review, handoff и rollback |
| снижение зависимости от чата | проект можно продолжать без привязки к конкретной сессии |
| предсказуемость масштабирования | десятки лендингов можно вести по одной модели |

У этого решения есть и дисциплинарная цена. Команда обязана обновлять документацию вместе с изменениями, а не постфактум. Локальные файлы, чат-заметки и временные артефакты могут использоваться в работе, но не должны оставаться единственным местом хранения значимой информации.

## Operational Rule

Если важное решение существует только в переписке, локальном markdown-файле или памяти исполнителя, значит для проекта его ещё не существует. Чтобы решение стало частью управляемой системы, оно должно быть зафиксировано в GitHub.
